# Самостійна робота №2

**LSP** стверджує: підкласи мають доповнювати, а не заміщувати чи анулювати поведінку базових класів.

---

### 1. Нелітаючий пінгвін (Проблема інтерфейсу)
* **Суть:** Клас `Bird` має метод `fly()`. `Penguin` наслідує його, але кидає `NotImplementedException`.
* **Чому порушення:** Клієнтський код, що очікує політ від будь-якого `Bird`, ламається при зустрічі з пінгвіном.
* **Рішення:** Розділити ієрархію. Винести `fly()` в окремий інтерфейс `IFlyable`.

### 2. Read-Only Користувач (Проблема стану)
* **Суть:** Клас `User` дозволяє змінювати пароль через `changePassword()`. `GuestUser` блокує цю дію (порожній метод).
* **Чому порушення:** Порушується "контракт" то система вважає дію виконаною, хоча пароль не змінився, що призводить до логічних помилок.
* **Рішення:** Спадкувати обох від базового `BaseUser` (тільки дані), а зміну пароля додати лише в `AuthorizedUser`.

### 3. Драйвер сховища (Проблема прав доступу)
* **Суть:** Базовий клас `Storage` має метод `write()`. Підклас `ReadOnlyStorage` ігнорує запис.
* **Чому порушення:** Викликаючий код не отримує підтвердження помилки, але дані не зберігаються. Порушується цілісність даних.
* **Рішення:** Розділити функціонал на інтерфейси `IReadable` та `IWritable`.

---

### Висновок
Порушення LSP зазвичай виникає, коли використовується **спадкування заради повторного використання коду**, а не за логікою поведінки. Якщо метод підкласу не може виконати те, що обіцяє батьківський клас то ієрархію побудовано невірно.
